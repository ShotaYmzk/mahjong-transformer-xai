# 作業記録の運用

作成日: 2026/04/29

今後は、READMEを大きくしすぎず、日々の成果や対処法を `docs/` に残す。

## 基本方針

READMEはプロジェクトの入口にする。

詳細な成果記録、手順、トラブル対応、今後の方針は `docs/` に置く。

作業したら、コード変更だけで終わらせず、必要な記録も一緒に更新する。

新しくMarkdownドキュメントを作る時は、先頭のタイトル直下に `作成日: YYYY/MM/DD` を書く。

既存のMarkdownを編集する時に作成日がなければ、内容を壊さずに同じ形式で追記する。

## 置き場所

日ごとの成果:

```text
docs/progress-YYYY-MM-DD.md
```

トラブル対応:

```text
docs/troubleshooting.md
```

モデル開発やデータ作成の手順:

```text
docs/model-development-guide.md
```

フォルダやファイルの役割:

```text
docs/project-structure.md
```

ビューアやUIの使い方:

```text
docs/viewer-guide.md
```

## 成果記録に書くこと

- 作成日。
- 何を実装したか。
- なぜ必要だったか。
- どのファイルを変更したか。
- どう確認したか。
- 未解決の問題。
- 次にやること。

## トラブルシューティングに書くこと

- 作成日。
- 症状。
- 原因。
- 修正内容。
- 確認方法。
- 再生成や再学習が必要か。
- 次に同じ問題が起きた時に見るファイル。

## READMEに書くこと

READMEには以下だけを書く。

- プロジェクトの目的。
- 今の状態。
- 最短セットアップ。
- 主要ドキュメントへのリンク。
- 重要な注意点。

日々の長い成果ログや詳細なコマンド履歴はREADMEに書かない。

## 記録更新のタイミング

以下の作業をしたら、関連docsも更新する。

- データ仕様を変えた。
- モデル構造を変えた。
- 学習スクリプトや実行方法を変えた。
- 新しい評価結果が出た。
- トラブルを直した。
- ビューアやレポート形式を変えた。
- 今後の作業方針が変わった。
